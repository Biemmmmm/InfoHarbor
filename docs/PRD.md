# InfoHarbor – 个人信息坞 / 信息操作系统 – PRD（V1）

## 1. 背景与问题

在 AI 和社交媒体高度发达的环境下，信息生产与传播速度极快：

- 各类模型、媒体、社交账号持续输出内容；
- 各种资讯源（RSS、新闻站点、Telegram 频道、X、博客、群聊等）高度碎片化。

典型用户（如技术从业者、投资者、内容密集型知识工作者）面临的共性问题：

1. **信息过载与认知负荷过高**
   - 每天要在多个 App 和网页之间来回切换；
   - 大量时间消耗在“找信息 + 滤垃圾 + 去重”上，而不是在真正创造价值和做决策。

2. **注意力分散**
   - 不同源的信息不断打断深度工作；
   - 没有统一地方管理“今天真正需要关注的少数重点”。

3. **学习与认知迭代缺乏结构化积累**
   - 看过很多信息，但缺少系统化的记录与反思；
   - 很难形成“从输入 → 思考 → 输出 → 再输入”的正向学习循环。

4. **个性化不足**
   - 现有资讯聚合产品难以真正理解“对我来说什么是高质量信息”，
   - 用户偏好、理解和判断标准很难沉淀进系统。

InfoHarbor 希望通过统一入口、自动处理与个性化学习，帮助用户从信息过载中“抽离出来”，把注意力和时间集中于高价值信息和高质量学习。

## 2. 产品愿景与核心目标

### 2.1 愿景（长期）

构建一个通用的、可个性化配置的**个人信息操作系统（Personal Information Operating System）**，具备以下能力：

- 统一汇总多源信息流，自动清洗、分类和摘要；
- 帮助用户降低信息噪音和认知负荷，减少注意力发散；
- 辅助用户记录自己的认知、观点和判断；
- 通过持续交互数据，拟合并更新“对该用户而言什么是高质量信息”，实现越用越好用；
- 支持多种行动域（如学习、研究、项目管理、交易等）的信息驱动决策。

### 2.2 V1 核心目标（MVP）

V1 仅为最小可用版本，聚焦于：

1. **统一入口 & 降噪**
   - 从有限数目的高价值源（例如 RSS + Telegram）自动拉取信息；
   - 将原始内容转化为结构化信息卡片（InfoCard），包括摘要、标签和简单评分。

2. **注意力管理**
   - 提供一个 Web 看板，用户每天/每周只需在一个面板上查看重点信息；
   - 实现基础的“优先级 + 注意力预算”（每天展示的卡片数量有限）。

3. **用户认知记录**
   - 用户可以在每条信息卡片上记录自己的想法、理解、问题和决策；
   - 系统将这些记录与原始信息关联起来，形成“信息 → 认知”的轨迹。

4. **信息质量与相关性评价**
   - 用户可以对信息卡片进行质量/相关性评价（评分/打标）；
   - 系统记录这些反馈，用于后续提升排序和过滤效果。

5. **个性化拟合（初级）**
   - 基于用户的评价和交互行为，统计不同信息源、主题、内容类型的“有用率”；
   - 为后续的智能排序和推荐预留数据与结构（V1 只做基础统计和简单规则调整）。

6. **行动域示例 – 交易信号解析（不执行交易）**
   - 对特定来源（如 Telegram 交易频道）的文本进行结构化解析；
   - 提取交易相关字段（品种、方向、价格区间等），以信息卡片中的子块形式展示；
   - 用于说明系统也可以支持“从信息到行动建议”的场景，但 V1 不接入真实交易执行。

## 3. 核心概念与数据模型

### 3.1 IPO 信息流模型

InfoHarbor 按照 Input – Process – Output 三个阶段组织系统：

- **Input（输入）**
  - 各类外部信息源：RSS、新闻 API、Telegram 频道等；
  - 以统一的原始事件（RawEvent）形式落地。

- **Process（处理）**
  - 标准化、去重、摘要、分类、打标签和评分；
  - 聚合为信息卡片（InfoCard）；
  - 周期性地对一段时间内的卡片做“高层总结”（PatternCard）。

- **Output（输出）**
  - 在信息看板（Dashboard）中展示有限数量的高价值信息；
  - 支持用户对卡片进行：
    - 阅读、筛选和排序；
    - 记录自己的想法和判断；
    - 进行质量/相关性评价；
  - 交互数据被系统记录，用于个性化拟合。

### 3.2 核心实体（概念级）

- **Source**
  - 一个外部信息源（如某个 RSS Feed、某个 Telegram 频道）。

- **RawEvent**
  - 从某个 Source 拉取的原始内容记录。

- **InfoCard**
  - 一条面向用户的“信息单元”，通常由一个或多个 RawEvent 聚合而成；
  - 包含摘要、标签、评分、来源列表等。

- **PatternCard（可选）**
  - 对一段时间内的若干 InfoCard 进行总结的“模式卡片”，用于复盘和高层学习。

- **UserInteraction / Feedback**
  - 用户在 InfoCard 上的操作：标记有用/无用、评分、保存稍后阅读、归档等。

- **UserNote**
  - 用户针对某条 InfoCard 写下的自由文本（认知记录、想法、问题、决策等）。

- **TradeSignal（行动域示例）**
  - 从交易类文本中解析出的结构化交易信号，用于展示和分析。

实体具体字段由后端在实现时根据本 PRD 细化。

## 4. 用户画像与场景

### 4.1 用户画像（V1）

V1 默认单用户，但从设计上要适合通用用户特征：

- 信息密集型工作者（如工程师、架构师、研究员、产品经理、投资者等）；
- 每天需要跟踪多个领域（技术、行业、宏观、市场等）的变化；
- 对学习和自我提升有较高诉求，愿意在信息之上做记录和反思；
- 希望通过工具降低信息噪音，提升决策与学习效率。

### 4.2 典型场景

1. **每日信息面板**
   - 用户每天打开 InfoHarbor，只需要看一个 Web 面板；
   - 系统已从多个源聚合、去重、摘要，将重点内容转为 InfoCard；
   - 面板中只展示有限数量的高优先级卡片。

2. **深度阅读与认知记录**
   - 用户打开某条 InfoCard 的详情页；
   - 看到原文/链接、摘要、标签等；
   - 在“我的笔记”区域记录自己的理解、判断和后续行动想法；
   - 这些笔记与卡片长期关联，可在未来复盘时使用。

3. **信息质量与个性化**
   - 用户在阅读卡片后，给出：
     - 质量评价（例如 1–5 分）；
     - 相关性评价（例如“对我很有用 / 一般 / 无用”）。
   - 系统统计不同源、主题、内容类型的“有用率”，为后续排序和过滤提供基础。

4. **主题/领域学习与复盘**
   - 用户按主题筛选某一领域的信息（如 AI 技术或宏观经济）；
   - 使用 PatternCard 或简单统计查看：
     - 近期有哪些重复出现的话题和观点；
     - 哪些信息被多次标记为“有用”。

5. **行动域示例：交易信号解析（不执行交易）**
   - 系统从交易频道的文本中识别交易信号，生成带结构化字段的 InfoCard；
   - 用户可以把自己的判断记录在 UserNote 中，并给出质量评价；
   - 后续可以基于这些信号和评价做手工或模拟分析。

## 5. V1 范围与非范围

### 5.1 V1 必须实现（MVP）

1. **信息源接入（首批）**
   - RSS / 技术与新闻源；
   - Telegram 频道（信息/资讯/交易类）；
   - 源配置通过配置文件或简单数据表管理。

2. **采集与标准化（RawEvent）**
   - 定时任务从各 Source 拉取内容；
   - 写入统一结构的 RawEvent 表；
   - 基本防重逻辑（基于源和外部 ID/URL）。

3. **InfoCard 生成**
   - 对 RawEvent 进行：
     - 摘要（短文本）；
     - 主题分类（如：ai-tech, web3, macro, market, learning, other）；
     - 初步重要度/优先级打分；
   - 简单的聚合（例如：相同 URL 或高度相似标题的内容合并为一个 InfoCard）。

4. **Dashboard 看板**
   - Web UI，黑色/深色主题；
   - 至少三个列：
     - Today（今日重点）
     - This Week（本周内可处理）
     - Background（背景信息或低优先级）；
   - 支持按主题、时间、来源过滤。

5. **注意力预算与展示控制**
   - 支持为 Today 列配置最大卡片数量（可以按主题域设置不同上限）；
   - 根据重要度打分，对超出预算的卡片自动分配到 This Week / Background 列。

6. **用户反馈与评分**
   - 用户可以对 InfoCard：
     - 标记有用/无用；
     - 标记稍后再看（移动到 This Week）；
     - 标记归档；
     - 进行简单评分（例如 1–5 分，用于质量/相关性）；
   - 所有反馈写入 Feedback / UserInteraction 表。

7. **用户笔记（认知记录）**
   - 在 InfoCard 详情中提供“笔记/评论”输入区域；
   - 用户可以记录自己的想法、理解、问题或决策；
   - 笔记长期与 InfoCard 关联，可在未来再次查看和搜索；
   - 后端保存 UserNote，与用户 ID 和 InfoCard ID 关联。

8. **交易信号解析（行动域示例，仅限记录）**
   - 对来自指定 Source 的交易文本调用提取逻辑：
     - 识别 symbol, direction, entry_range, stop_loss, take_profit 等；
   - 在 InfoCard 中展示结构化的交易信号块；
   - 将信号保存到 TradeSignals 表，以供后续分析；
   - 不实现任何真实交易执行。

### 5.2 明确不在 V1 范围的内容

- 真实资金交易执行、交易所 API 接入；
- 多用户系统（V1 假设单用户环境，设计上保留扩展空间即可）；
- 复杂的个性化推荐/排序模型（如协同过滤、大规模训练）；
- 移动原生 App（可通过响应式 Web 或 PWA 暂时代替）。

## 6. 非功能性需求（NFR）

1. **通用性**
   - 文档和实现不要依赖用户的个人世界观或专有术语，适合不同类型用户使用；
   - 主题和标签体系可配置。

2. **扩展性**
   - 信息源接入通过统一接口，支持后续添加新源；
   - 数据结构预留字段用于存放更多元数据和算法结果。

3. **可维护性**
   - 代码模块化清晰：sources、processing、cards、attention、feedback、notes、trading_signals 等；
   - 至少对核心数据模型和 API 进行基础测试。

4. **安全**
   - 不在代码库中存储真实 API Key/Token；
   - 不处理真实交易执行操作。

## 7. 技术栈与架构建议（同前）

- 后端：Python + FastAPI + PostgreSQL +（可选）Redis/队列；
- 前端：Next.js + React + TypeScript + Tailwind；
- 基础设施：Docker / docker-compose。

高层架构：

- 收集层：Ingestion Service → RawEvents；
- 处理层：Processing Service → InfoCards / TradeSignals / （可选）PatternCards；
- API 层：REST API；
- 展示层：Dashboard（看板 + 详情 + 搜索与过滤）。

具体细节请在 `ARCHITECTURE.md` 或 `API.md` 中进一步补充。
