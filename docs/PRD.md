# InfoHarbor – 个人认知负熵系统 / 信息坞 – PRD（V1）

## 1. 背景与本质问题

在 AI 时代，信息生产和传播的速度被大幅加速：  
- 模型、媒体、社交平台源源不断地产生内容；
- 各种 KOL、机器人账号和自动化系统也在持续放大信息噪音。

结果是：**信息熵指数级膨胀**，而人类的大脑带宽和注意力是极其有限的。

当前状态下，我需要：

- 作为架构师 / 创业者：持续跟踪技术趋势、工具栈演进、架构思想；
- 作为投资者 / 交易者：关注宏观局势、资产价格、资金流动与多源信号；
- 作为自我迭代的个体：持续学习新方法论，总结经验，升级自己的“道、法、术、器”。

现实问题是：

1. 信息源极度碎片化：  
   - Telegram 频道、X、RSS、新闻网站、博客、视频、课程、群聊……
2. 注意力被大量磨损在“找信息 + 过滤垃圾 + 来回切 App”上，  
   - 而不是“做抽象、找规律、升级认知模型”上。
3. 信息层级混乱：  
   - 道 / 法 / 术 / 器混在一起，导致容易沉迷于“术与器”的细节而忽视更本质的规律与方法论。
4. 认知迭代不成系统：  
   - 看了很多，但难以形成稳定的模型库与可复用的行动框架；
   - 容易出现“garbage in, garbage out”的认知消耗。

**第一性原理：**

> AI 时代加速了外部世界的信息熵增，  
> InfoHarbor 的使命是充当我的“麦克斯韦妖 / 认知负熵机器”，  
> 在输入层面筛选 & 压缩，在处理中提升信息密度，在输出端引导更高层次的“道-法-术-器”认知与行动。

## 2. 产品愿景与核心目标

### 2.1 产品愿景（长期）

构建一个**个人认知操作系统**（Cognitive OS）：

- 它是我的**信息坞**：所有重要信息流统一收口；
- 它是我的**负熵机器**：把杂乱无章的信息变成高密度、可直接“食用”的认知和行动建议；
- 它是我的**认知 IPO 循环引擎**：
  - Input：持续、高质量、自动化地获取多源信息；
  - Process：通过 LLM 和规则，把信息加工成结构化卡片、抽象模式与方法论；
  - Output：帮助我做更优决策、迭代认知模型、沉淀“道-法-术-器”。

### 2.2 V1 核心目标（最小可用）

V1 不追求“覆盖所有场景”，而是聚焦在**几个认知基础功能**上：

1. **降低认知负荷（减少信息摩擦）**
   - 将多个高质量信息源统一到一个面板；
   - 自动摘要、分类、打标签，避免我重复打开 N 个 App 过滤垃圾。

2. **减少注意力发散**
   - 用“看板 / 列表 + 注意力预算”的方式，只推送有限数量的高价值信息卡片；
   - 避免无休止的刷屏行为。

3. **成倍提高效率**
   - 让大脑直接“食用”已经压缩过的信息，要么是高度摘要，要么是抽象模式；
   - 对细节内容提供“按需展开”，而不是被动被细节淹没。

4. **持续迭代认知，发现本质规律**
   - 不只帮我“看更多”，而是帮助我从信息中提炼：
     - 框架 / 方法论（法）
     - 认知升级点 / 世界观修正（道）
   - 支持简单的“模式提取 / 周期性复盘”：让系统把多条信息的共性抽出来。

5. **提供“道-法-术-器”多层指导和工具**
   - 对每条信息（或一组信息）尝试给出：
     - “这是哪一层的信息？”（道/法/术/器）  
     - “在各层我应该如何行动/思考？”

6. **自动化获取信号与执行只是众多输出之一**
   - 自动化交易信号解析、策略建议、操作引导，是“术/器层”的一个典型应用；
   - 它服从于更高目标：**不牺牲认知质量的前提下，提高行动效率**。

## 3. 核心概念与模型

### 3.1 认知 IPO 循环

- **Input（输入）**
  - 来自多个信息源：技术、宏观、市场、项目、个人兴趣等；
  - 偏重“高质量源 + 结构化源”，减少纯噪音流。

- **Process（处理）**
  - 信息清洗、标准化、摘要、分类；
  - 打“道-法-术-器”层级标签；
  - 识别模式：相似事件、重复观点、趋势变化；
  - 生成“认知卡片”（InfoCard）和“模式卡片”（PatternCard）。

- **Output（输出）**
  - 给我一个有限大小的“今日 / 本周认知面板”；
  - 输出：
    - InfoCard：具体信息和可执行建议；
    - PatternCard：抽象出的规律 / 模型 / 方法论；
    - 行动建议：包括但不限于技术实践、学习计划、交易模拟等。

### 3.2 道-法-术-器层级

对每条信息，系统尝试判断其所在层次：

- **道**：世界观、价值观、第一性原理、长期趋势、底层假设。
- **法**：可复用的方法论、框架、策略（例如：某种交易/研究/学习方法）。
- **术**：具体的技巧、实操步骤、pattern、tips。
- **器**：工具、平台、具体实现、命令、代码片段等。

目标：

- 在界面上允许我按层级过滤：
  - “只看法 & 道”，进行高层认知升级；
  - 或“只看术 & 器”，进行战术执行；
- 系统在生成 PatternCard 时，优先把“法 / 道”层内容提炼出来。

## 4. 用户画像与场景

### 4.1 用户画像（V1）

- 技术创业者 / 架构师；
- 主动交易者，关注多类资产与策略；
- 极度注重“认知升级”和“时间 / 注意力 ROI”；
- 已具备一定的自我框架体系（如：IPO、Y-模型、道法术器），希望系统能与这些元模型协同。

### 4.2 核心使用场景

1. **认知雷达（Cognitive Radar）**
   - 我每天打开 InfoHarbor，只需要看一块面板；
   - 面板上已经整合了技术、宏观、市场、个人项目等多维信息；
   - 每张卡片已经是“高度压缩 + 结构化”的信息。

2. **认知蒸馏（Insight Distillation）**
   - 对一定时间窗口内（例如过去一周）的高价值信息，系统自动生成“模式卡片”：
     - 近期市场行为出现了哪些重复模式？
     - 技术演进中有哪些隐含路径？
     - 哪些观点在多个源被重复出现，值得纳入“法/道”层？

3. **道法术器视图**
   - 我可以切换视图：
     - “只看法与道” → 获取框架与原则；
     - “只看术与器” → 获取具体操作建议/工具链升级。

4. **注意力控制与预算**
   - 每天系统限制推送的 InfoCard 数量（按主题域 & 层级）；
   - 避免我被细节淹没，只在有限时间处理最关键的信息与模式。

5. **行动域（包括但不限于交易）**
   - 对于某些领域（如交易、项目决策、学习计划），系统在卡片中给出“建议行动”：
     - 例如：  
       - 交易：仅在 V1 阶段给出结构化信号与模拟建议；  
       - 技术：给出“实践此技术的最小实验”建议；  
       - 学习：给出“下一步精读/练习”的建议。

## 5. V1 范围（Scope）

### 5.1 必须实现（MVP）

1. **信息源接入（首批）**
   - RSS / 技术博客 / 新闻源（技术 + 宏观）
   - Telegram 频道（技术/信息类 + 交易/市场类）
   - 预留接口扩展其他源（X、GitHub Releases 等），但不要求 V1 立即实现。

2. **统一采集与标准化（RawEvent）**
   - 采集任务统一写入 `RawEvents` 表：
     - source_type, source_id, fetched_at, raw_content, metadata
   - 基本去重逻辑：按 source_id + external_id/URL。

3. **InfoCard 生成管道**
   - 对 RawEvent 进行：
     - 摘要（1–3 句）
     - 主题域分类（例如：`ai-tech`, `web3-tech`, `macro`, `market`, `learning`）
     - 道-法-术-器层级判定（至少初版基于 LLM+规则）
     - 重要度评分（与个人关注域匹配度 + 来源可靠度 + 时效性）
   - 聚合多条相关 RawEvent 成一个 InfoCard（简单规则即可）。

4. **PatternCard（模式卡片）基础**
   - 在一个时间窗口内（例如最近 N 条高优先级 InfoCard），  
     - 调用 LLM 生成一个“模式总结”：
       - 近期重复出现的主题 & 观点；
       - 潜在的规律 / 方法论（优先标为“法/道”）。
   - PatternCard 是一种特殊类型的卡片，展示在“模式 / 复盘”区域。

5. **认知看板（Dashboard）**
   - 以看板方式展示 InfoCard：
     - 至少 3 列：
       - Today（今日重点）
       - This Week（本周内消化）
       - Background（背景信息）
   - 支持按：
     - 主题域（技术/宏观/市场/学习/项目）
     - 道法术器层级
     - 时间范围
     进行筛选。

6. **注意力预算与卡片分配**
   - 配置每日/每主题域的最大 InfoCard 数量（Today 列）。
   - 按 `priority_score` 排序，超出预算的卡片自动降级到 This Week / Background。

7. **反馈机制**
   - 用户可对 InfoCard 进行操作：
     - 有用（useful）
     - 无用（useless）
     - 稍后再看（move to This Week）
     - 归档（archive）
   - 后端记录 Feedback 数据，为后续“权重调整 / 源质量评估”提供依据。

8. **交易信号解析（仅为行动域示例，不执行交易）**
   - 对交易相关文本（如指定 Telegram 频道）：
     - 解析 symbol, direction, entry, stop_loss, take_profit 等。
   - 在 InfoCard 中以“交易信号”子块展示。
   - 写入 `TradeSignals` 表，供未来回测/策略研究使用。
   - 不实现真实账户的下单/撤单执行。

### 5.2 明确不在 V1 的内容

- 实盘自动交易执行（包括调用交易所 API 下单）。  
- 多用户体系与权限管理（V1 仅单用户：我本人）。  
- 复杂的个性化推荐/排序模型（V1 使用规则 + 简单分数即可）。  
- 完整移动 App（可以通过响应式 Web 或 PWA 暂时代替）。

## 6. 功能需求细化（V1）

### 6.1 信息源管理

- 可通过配置文件或简单管理界面维护 Source：
  - `id, type, name, config_json, enabled, priority_weight`
- 支持启用/禁用某源。

### 6.2 原始事件采集（Ingestion）

- 定时任务（如每 5 分钟）拉取启用 Source 的最新内容。
- 将每条内容标准化为 RawEvent 写入数据库。
- 处理错误与重试，不因单一源错误阻塞整体任务。

### 6.3 InfoCard 生成

- 对新 RawEvent 启动后台任务：
  - 调用 LLM：
    - 生成摘要；
    - 判定主题域；
    - 判定道法术器层级；
    - 判断是否与已有 InfoCard 聚合；
  - 计算初始 `priority_score`。
- 支持 InfoCard 与 RawEvent 多对多关联。

### 6.4 PatternCard 生成

- 支持按时间窗口或数量阈值，定期生成 PatternCard：
  - 输入：一段时间内的高优先级 InfoCard。
  - 输出：
    - 若干关键模式/趋势的总结；
    - 对应的“法/道”层结论；
    - 简短的“如何在下周调整认知/行动”的建议。
- PatternCard 在 UI 中单独区域展示，可用于每周复盘。

### 6.5 Dashboard UI

- 多列看板布局，黑色主题。
- InfoCard 显示：
  - 标题（AI 生成）
  - 摘要
  - 来源列表 & 时间
  - 主题域标签
  - 道法术器层级标记
  - 重要度/可信度简要指示
- 详情视图：
  - 原文 / 链接
  - 关联 RawEvent 时间线
  - 更长的总结与洞察
  - 道法术器拆解（系统尝试给出：这条信息属于哪层、如何使用）
  - 反馈操作按钮

### 6.6 Feedback 与简单学习

- Feedback 表记录：
  - user_id, infocard_id, action_type, created_at, metadata
- 提供简单统计：
  - 每个 Source 被标记“无用”的比例；
  - 每个主题域/层级的“有用率”；
- V1 不必须自动调整源权重，但为后续算法预留字段 `source.priority_weight` 等。

## 7. 非功能性需求（NFR）

1. **认知友好性**
   - UI 避免信息过载；
   - 默认展示视图偏“精简”，需要时才展开细节。

2. **扩展性**
   - Source 接入通过统一接口，允许未来添加更多源；
   - InfoCard / PatternCard 数据模型需支持增加字段与标签。

3. **可维护性**
   - 后端模块化（sources / processing / cards / attention / feedback / trading_signals）。
   - 有基础测试覆盖 InfoCard 生成逻辑、PatternCard 生成逻辑。

4. **安全**
   - 不实现真实交易；
   - 外部 API Key/Token 通过环境变量配置，不写入仓库。

## 8. 技术栈与架构建议

与之前一致，建议：

- 后端：Python + FastAPI + PostgreSQL + （选）Redis/队列
- 前端：Next.js + React + TypeScript + Tailwind（深色主题）
- 基础设施：Docker / docker-compose

架构层次：

- 收集层：Ingestion Service → RawEvents
- 处理层：Processing Service → InfoCards / PatternCards / TradeSignals
- API 层：REST API → 前端
- 展示层：Dashboard Web App（看板 + Pattern 视图）
